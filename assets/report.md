## 技术分析 / Technical Analysis

### 排查过程 / Investigation Process

本节用于记录从告警线索到结论形成的排查过程。写作时按实际调查顺序展开：先说明触发排查的告警、客户反馈或异常指标，再说明核验过的数据源，例如云审计、登录日志、EDR、主机进程、网络连接、文件落地记录、安全组与云防火墙命中记录。每一步都要写清楚观察到的事实、对应证据、当时形成的判断，以及为什么继续沿该方向排查。

推荐写法是按证据链分段：第一段写初始线索和确认方式；第二段写账号、入口或漏洞如何被定位；第三段写攻击者在云侧或主机侧做了什么；第四段写哪些行为被证实、哪些行为被排除。不要只堆 ATT&CK 技术名，需要把关键时间、主体、资源、命令、IP、哈希和日志来源写清楚。未观察到的行为也要说明证据边界，例如未发现权限提升、持久化成功、数据外传或防御设施被篡改。

### 样本分析 / Sample Analysis

若事件中存在恶意文件、脚本、WebShell、挖矿程序、扫描器、隧道工具或持久化组件，本节按样本逐个说明。每个样本至少写明文件名、路径、大小、哈希、文件类型、发现位置、落地时间和关联进程。若没有样本，保留本节并说明未发现样本或样本不可获取，不要凭空补写。

分析正文要写行为和结论：样本是否具备扫描、提权、远控、隧道、窃密、加密、挖矿或日志清理能力；是否有外连地址、命令参数、配置文件或计划任务；它与本次事件中其他证据如何关联。不要只贴哈希，也不要把样本家族名当作唯一结论。

### 入侵路径 / Intrusion Path

本节用于把排查结论整理成一条可阅读的路径。先写入口或初始访问方式，再写横向或控制面动作，随后写主机侧执行、样本落地、外连、持久化、收集或外传尝试，最后写抑制动作切断了哪一段链路。若插入图片，图片下方仍要保留文字说明，文字中标注关键时间、源 IP、账号、实例、命令、文件和告警点。

## 响应行动 / Response Actions

::: phase 抑制阶段 / Containment
抑制阶段只写已经执行或正在执行的止损动作，目标是阻断攻击者继续操作。状态标记含义：[x] 已完成，[/] 进行中，[ ] 未开始。

- [x] 写明已完成的隔离、封禁、下线、吊销会话、停用账号、阻断外连或停止恶意进程动作。
- [/] 写明仍在推进的抑制动作，以及卡点、责任方和预计完成时间。
- [ ] 写明尚未执行但需要客户确认的止损动作，避免写成已经完成。
:::

::: phase 根除阶段 / Eradication
根除阶段写清如何移除攻击者遗留能力和根因条件。不要把建议写成事实，已经完成的动作和待客户执行的动作要分开标记。

- [x] 写明恶意文件、计划任务、后门账号、公钥、WebShell、异常服务或云侧策略是否已清理。
- [/] 写明镜像重建、补丁修复、密钥轮转、权限回收、漏洞修复等仍在推进的动作。
- [ ] 写明需要客户继续确认的根因修复，例如源码仓库排查、凭证治理、组件升级或业务改造。
:::

::: phase 恢复阶段 · 处置建议 / Recovery & Hardening
恢复阶段写长期加固建议，目标是降低同类事件复发概率和影响半径。建议按风险优先级排序，不要堆通用安全清单。

- [ ] 身份与权限：写明是否需要禁用主账号 AK、改用 RAM 角色、收敛高权限策略、配置访问白名单、建立密钥轮转周期。
- [ ] 暴露面：写明需要关闭哪些公网入口、哪些服务应加白名单、哪些端口或域名应接入 WAF、云防火墙或安全组策略。
- [ ] 漏洞与配置：写明需要升级的组件、修复的漏洞、加固的中间件、需要复查的同类系统。
- [ ] 可观测性：写明需要开启和留存哪些日志，例如云审计、云安全中心、WAF、主机登录、进程启动、网络连接和应用访问日志。
- [ ] 演练与治理：写明是否需要渗透测试、红蓝对抗、凭证泄露扫描、资产盘点、基线检查或定期审计。
:::

## 总结与安全建议 / Summary & Advisory

### 事件总结 / Event Summary

事件总结用于收束全文。第一段复述最终定性、影响范围、处置状态和残余风险，语气要确定，不能引入前文没有支撑的新结论。

第二段写根因和经验教训，重点说明为什么会发生、哪类控制失效、哪类检测有效。不要写泛泛的提高安全意识，要对应本次事件里的证据。

第三段写后续优先级，把恢复阶段建议压缩成几条客户能执行的主线，例如身份治理、暴露面收敛、日志留存、漏洞修复和检测规则建设。

如果仍有未闭环风险，在本节最后明确复评条件：需要哪些日志、哪些整改证明、哪些扫描或演练结果，才能把残余风险从高或中降到低。
