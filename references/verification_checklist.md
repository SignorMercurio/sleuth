# 结论对抗式验证清单（出报告前质检门）

SKILL.md 步骤 7 在生成报告前加载本文件。目的：把报告里每条「承重断言」交给独立核验后再定稿，系统化防止**误归因**——本领域最大的报告风险。清单条目本身来自各 `invest_*` / `tech_*` 指南里散落的判读注意事项，这里集中成一道强制门。

## 承重断言（必须逐条过门）

会写进报告、且支撑定性或处置的结论：

- 事件定性（是什么攻击 + 是否得手）
- 入侵路径的每一步（入口 / 执行 / 外联 / 持久化 / 处置）
- `::: attack` 矩阵每个点亮的技术
- 每条「攻击者做了 X」的行为结论
- 每个 IoC 的归因（某 IP / 文件 / 进程属于攻击者）
- 时间线的关键节点（T0、止损点）

非承重的过程描述、「未观察到」类陈述不必逐条过门。

## 核验方式

1. 列出全部承重断言。
2. 逐条（断言多时按报告章节分组）派**独立核验 agent**：
   - 运行时支持子 agent 就派（见 SKILL「运行环境兼容」的派生子 agent 映射）；只给该 agent「断言 + 所引证据 + 本清单」，**不给主 agent 的推理过程**，避免确认偏误。
   - 运行时不提供子 agent，则主 agent 换「我要推翻这条」的怀疑视角逐条自检。
   - 核验 agent 默认存疑，任务是**尽力反驳**；必要时可跑少量定向只读命令补证（如查中间件配置、日志覆盖窗），但不重做整轮采集，且同受只读护栏约束。
3. 裁决并改写报告措辞：
   - 被成功反驳 / 仅孤证 / 证据不足 → 该结论在报告里降级（「已确认」→「推测」/「无法确认」/「未观察到」）或删除。
   - 扛住反驳 → 保留确定措辞。
4. 产出核验表（内部用，不进报告）：`断言 | 裁决 | 反驳依据 | 报告措辞调整`。

## 反驳清单（逐类质疑）

**时间证据**
- `atime` 不等于攻击者访问——云安全中心 / SAS 扫描、备份、杀软读取都会刷新；要有 Web 日志或进程遥测交叉。
- `mtime` 很早但 `ctime` 很新 = 时间戳可能被篡改，别直接采信 mtime。
- `-newermt` 时间窗命中的文件需人工研判，不是命中即恶意。

**存在 ≠ 利用 / 得手**
- WebShell / 后门文件存在 ≠ 当前可用：查中间件配置（如 `location ~ \.php$ { return 403; }`）+ 实际访问状态码。
- SUID / 可疑二进制存在 ≠ 已被用于提权：要有调用证据。
- 文件落地 ≠ 攻击成功：区分「上传成功」与「利用成功」。

**IP / 来源归因**
- 应用 / Nginx 日志里是反代 IP，WAF 看到的才是真实攻击者 IP。
- 对 WebShell 目录后来的 GET / 访问 ≠ 原始上传源 IP，只能写「近期访问 / 探测」。
- IP 归属与信誉要外部情报佐证，别凭地域臆断。

**云端措辞**
- WAF `waf_action='block'` ≠ 实际拦截：结合 `waf_test` / `final_action` / `upstream_status` / `status` 判断，test / 观察模式不是真拦。
- 「未检出」≠「未发生」：必须核对日志覆盖时间窗是否早于攻击窗口；遥测起始晚于攻击窗时只能写「无法据此排除」。

**OOB / DNSLog**
- DNS / OOB 域名命中 ≠ 主机失陷：需后续命令执行、外联、落地文件等跟进证据才能升级定性。

**因果与孤证**
- 进程父链 / 时间相邻是巧合还是因果？父链命中要能解释行为。
- 每个点亮的 ATT&CK 技术要有具体证据，不能因为「这类攻击通常会」就点亮。
- 关键结论至少两类独立证据源（日志命中 / 命令记录 / 样本行为 / 网络连接 / 客户确认）；孤证降级为推测。

## 与报告的衔接

核验后的措辞直接对应 `assets/report.md`：不确定写「未观察到 / 无法确认」，覆盖边界写进「覆盖说明」，残留风险只列证据支持的项。
