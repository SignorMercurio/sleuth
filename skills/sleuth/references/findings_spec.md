# Findings 中间文件规范（调查 → 报告的交接层）

每台主机完成步骤 2–7（含对抗式验证）后，把定稿结论落为一份 findings 文件；步骤 8 写报告时**只以 findings 文件为事实来源**，调查上下文中未写入 findings 的内容一律不得进入报告。这一层同时服务三个目的：多主机合并的输入、报告写作（含子 agent）的自包含输入、防止未经验证的结论渗入报告。

## 文件命名

```
IR-{YYYYMMDD}-{hostname}-{event_type}[-{event_id}].findings.md
```

字段含义与 `references/report_naming.md` 一致，生成在当前工作目录，单主机也生成（N=1）。findings 是工作底稿不是交付物：交付物只有最终一份 `IR-….md` 报告，报告交付后 findings 文件保留备查，去留由用户决定。

## 内容结构

按以下固定章节填写；没有内容的章节写「无」，不要删除章节（多主机合并时按章节对位读取）：

1. **主机标识** — hostname、Client ID、公网/私网 IP、实例 ID、资产用途
2. **事件定性** — 一句话定性 + 措辞等级（已确认 / 推测 / 无法确认），含主要 ATT&CK 技术编号
3. **时间线** — 4–6 个决定性节点，每条注明证据来源与时区
4. **入侵路径** — 按攻击阶段写已验证的步骤，每步附关键证据（命令 + 最短输出片段）
5. **ATT&CK 映射** — `T编号 :: 行为 :: 证据` 列表，只列通过步骤 7 验证门的
6. **IoC 清单** — IP / 域名 / 文件路径 / 哈希 / 账户，记录**原始值**（展示层转义在报告里做）
7. **漏洞** — 类型、受影响组件与版本、CVE（如有）、修复方案来源
8. **残留风险** — 步骤 6 六个维度的结果，逐项写「已清 / 仍在 / 未观察到」
9. **云端证据覆盖** — 日志源、查询时间窗、最早/最新记录、覆盖限制；未查云端时写明原因
10. **验证结果** — 步骤 7 核验中被降级或删除的断言及原因
11. **证据缺口** — 想查而没查到的项，是报告「覆盖说明」的素材

## 写作层使用规则

- 写作（步骤 8）只读：全部 findings 文件、报告模板、`references/report_naming.md`、`references/report_style.md`、`references/report_writing_rules.md`、本文件。不回看调查上下文，不碰 SIREN。内联写作且 findings 刚由本上下文写出时，无需回读文件。
- 写作不得改变事实与措辞等级：「推测」不得升级为「已确认」，IoC 不得增删改，时间不得四舍五入到改变先后关系。
- 写作中发现 findings 有缺口：不补写、不脑补，在报告对应位置写「无法确认」，并把缺口回报给编排者，由编排者决定是否回到步骤 2–7 补查。

## 多主机合并规则（多主机委托与合并模式）

合并模式（输入是用户提供的既有 `IR-….md` 报告）下，报告正文视作 findings 等价物，本文件全部规则同样生效：只做重组与改写表述，不得新增事实或升级措辞。

- **排查对象**：每台主机一个 `::: asset` 块（模板原生支持多资产）
- **时间线**：合并为一条统一时间线，节点标注主机名，节点数量上限同报告时间线约束
- **事件定性**：定性针对整个事件；各主机结论明显无关联（无共享 IoC、时间窗、手法）时仍只出一份报告，但在定性中分别陈述并写明「未发现关联」
- **ATT&CK 矩阵**：一个矩阵，任一主机有证据支撑即点亮该技术
- **技术分析**：按主机分小节，另加一节「跨主机关联分析」收纳横向移动 / 同源判断（须已过验证门）
- **响应行动**：合并去重，每条动作标注适用主机；全局动作（如封禁攻击 IP）只写一次
- **输出文件名**：按 `references/report_naming.md` 的多主机规则生成
