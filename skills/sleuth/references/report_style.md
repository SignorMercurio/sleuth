# 报告写作风格指南

步骤 8 写报告前读取本文件。目标：报告读起来像一名资深应急响应工程师写的，而不是模型生成的。下面的规则从作者手写文章（存放于 `assets/style/`）蒸馏而来；规则与样本冲突时以样本为准。

## 风格语料的使用

- `assets/style/` 存放作者手写的文章 / 报告样本（存放方式见该目录 README）。写作前列出该目录，**最多挑 2 篇**找语感；超长样本（约 300 行以上）只读最能代表行文的章节，不必全文读完。样本体量大时优先把写作交给子 agent，避免样本占满编排者上下文。目录为空就只按本文件的规则写。
- 样本是文风参照，不是事实来源：样本里的案例、命令、IP、结论一律不得进入报告。
- 样本有增删后，本文件的蒸馏规则需要人工重新核对一次。

## 句法与叙述

- **结论先行，推理透明**。先给判断，紧跟依据：「从告警详情的文件路径中，可以推测出……」。每个判断都交代它建立在哪条证据上，让读者能沿证据自己走到结论。
- **用完整段落叙述，列表只用于枚举**。叙述证据链时写成「检查了 X，发现 Y，因此 Z」的连续句；并列的方法、线索、待办与模板自带的骨架列表用列表呈现，每个列表项是完整的动作或判断，不是两三个词的短语。
- **长短句交替的自然书面语**。允许轻度口语化的转折与让步：「遗憾的是」「好在」「不难发现」「这并不总是那么容易」。不堆四字格，不写排比。
- **不确定性如实标注，且双向校准**。把握程度直接用「可能 / 推测 / 大概率 / 疑 / 未必」写进句子，证据不支持时写「无法确认 / 未观察到」；但**不要把证据充分的判断过度对冲**成「待进一步确认」——多条证据指向同一结论就写「大概率」，别退回「无法确认」。标注靠句中措辞，**不要用 `[推测]` 这类方括号标签，也不要另起一段写「本结论属高置信度推测，依据是……」的置信度元说明**——那是工作底稿写法，客户报告里把判断和依据写成一句通顺的话即可。
- **数字与范围具体**。「缩小至 3 项以内」「18:18:12 前后 ±1 秒」；可量化的不写「大量」「多次」。
- **证据贴身解释**。日志、命令输出引用后立即用一句话讲它说明了什么（「这条记录告诉我们……」），不让读者自己猜引用的意义。
- **中英文之间留空格**，专有名词用原文（WAF、ECS、Webshell），标点用全角中文标点。面向管理层的概述/结论里阿里云产品用中文名（云安全中心、云防火墙），`SAS`/`SLS`/`ActionTrail` 等缩写留给技术分析里的日志源与标识符上下文。

## 反 AI 腔清单（出现即改写）

- 三段式 / 排比式结构（「不仅…而且…更…」「无论是…还是…亦或…」）
- 填充性短语：「值得注意的是」「需要指出的是」「综上所述」「总而言之」「众所周知」
- 空洞评价词：「全面深入的分析」「高度复杂的攻击」「极大的安全隐患」——换成具体事实（复杂在哪、隐患是什么）
- 「首先 / 其次 / 再次 / 最后」式机械连接连续开段
- 每段一句话、通篇 bullet 的碎片化排版
- 正文里的「术语 :: 解释」腔与破折号滥用（模板 `:::` 指令块内的字段语法除外）
- 夹带非术语英文口语（hands-on、lateral movement、随手中英混写的 payload）——技术术语用通行中文或首次出现时括注，口语化英文直接换成中文（hands-on → 手动操作）
- 句尾拖「按 X 原则做 Y」的过程尾巴（「按推测记录并交叉其他证据」「按证据驱动原则不点亮该技术，仅在证据缺口里说明」）——判断和依据给完就收，与前面的置信度元说明同理，都是工作底稿口吻
- 图注复述正文（把相邻正文的时间、要素、限定语抄进图注）——图注一句话只标注图的对象，推理留正文
- 同句叠双层免责（「只表示 X，不能排除 Y」，而 Y 只是 X 的换个说法）——限定留一层就够

## 非通行措辞清单（出现即改写）

用词以国内安全工程师与客户口头交流的通行说法为准。自检：句子念出来，客户会停下来想「这个词什么意思」的，一律改写。典型四类：

- 英文行话直译的生造词：「扇出」「机群」「排查阴性」→「向多台主机发起 RDP 连接」「各 Windows 主机」「未发现后门」。没有通行中文译法的术语保留英文原词，不自造译名。
- 文言 / 公文字眼：「灭失」「3 年余」「且设永不过期」「史上仅此一次」→「已被覆盖丢失」「三年多」「并设置为永不过期」「此前从未出现过」。
- 整句压成的名词串：「入站鉴权全量基线中该来源史上仅此一次」「勒索工具包源」→ 拆回带动词的句子：「检索全部入站认证记录，该来源只出现过这一次」「存放勒索工具包的压缩包」。定语叠到两层就拆句，不生造复合名词。
- 比喻替代平实陈述：「精确套住」「压倒性主力」→「恰好覆盖」「绝大多数（336 次）来自堡垒机」。事实用平实动词加具体数字说，不靠修辞加重语气。

## 在报告各章的落点

- **事件概述**：用叙事段落讲清楚发生了什么，管理层 1 分钟能读完。
- **技术分析**：按调查的真实推理顺序写——观察 → 假设 → 验证 → 结论，让读者看见为什么排除了其他解释。
- **响应行动**：每条都以具体动词开头、有明确对象（「封禁 1.2.3[.]4 的入站访问」），不写「加强安全意识」这类没有执行主体的泛泛建议。
